## Decisions Made

- **In-process OpenAPI snapshot regen:** The dev-stack uvicorn at `:8000` was running an older build without postcard routes; calling `build_app().openapi()` directly via the venv Python guarantees the snapshot reflects the current source tree. This is the same pattern Plan 22-09 used for its FT-08 frontend integration and avoids needing to bounce uvicorn for every plan that lands new routes.
- **Form sends `address_block: null` (not omitted) when toggle is off:** Pydantic `Optional[AddressBlock] = None` accepts both undefined and null; explicit null mirrors the brochure pattern's `generate_images: false` boolean and surfaces the absence cleanly in the JSON body for tests + logging.
- **Empty-string -> null normalization in mutationFn:** `image_hint` and `brand_kit_slug` default to empty strings (per `useForm.defaultValues`); the mutationFn calls `.trim()` and sends `null` for empty values so the backend's `Optional[str] = None` is preserved (avoids 422 on the empty-but-present case where Pydantic sees `image_hint: ""` and rejects min_length).
- **`brand_kit_slug` zod refinement:** Schema is `.regex(SLUG).optional().or(z.literal(""))` — the `.or(z.literal(""))` is critical because the form ships an empty default. Without it, an unmodified form would fail validation due to the regex.
- **Test count:** 5 (plan asked for >= 4). The 5th test is end-to-end POST body capture, which verifies the full mutationFn wiring (especially the `address_block: null` normalization and the `template === "classic_portrait"` default). Field-render tests alone would not catch regressions in the optional-field normalization logic.
- **Editorial header test assertion strategy:** PageHeader renders `number` + slash + `kicker` as 3 separate `<span>` elements. Asserting on the composed string `"08 / THE MAIL"` would fail because no single text node contains that string. The test asserts on each piece separately (`getByText("08")` + `getByText(/the mail/i)`) plus the h1 heading.
- **Status page kicker mirrors brochure pattern:** The status page renders its own `08 / Job · {id-prefix}` kicker block (not via `<PageHeader>`) because the brochure pattern uses a custom h1 with the postcard-specific copy ("Postcard job") and the back-link to /postcards/new. Verbatim mirror of brochures/status.tsx.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] zod schema for `brand_kit_slug` rejected empty default value**

- **Found during:** Task 2 — running `pnpm test --run src/pages/postcards/` initially.
- **Issue:** The plan's interface block showed `z.string().regex(SLUG).max(64).optional()`. Combined with `useForm.defaultValues = { brand_kit_slug: "" }`, the regex check against an empty string fails before `.optional()` can short-circuit (zod's `.optional()` only allows `undefined`, not empty string).
- **Fix:** Added `.or(z.literal(""))` to the `brand_kit_slug` schema field. The mutationFn's empty-string-to-null normalization downstream still ensures the POST body sends `null` for empty values.
- **Files affected:** `frontend/src/pages/postcards/new.tsx` (line 64).
- **Committed in:** Task 1 (page scaffold) — caught during initial test run before commit.

**Total deviations:** 1 — minor schema fix to handle the empty default. No code-side scope creep; matches the locked PostcardCreateRequest backend shape.
