<objective>
Fix PLF-02: brochures render the literal hardcoded string "ESTATE PLANNING" as
the front-cover kicker even when the brief specifies a different tagline, AND
the perception loop reported that supplied `content.sections` were not visibly
rendered. The locked decision (per 24.1-CONTEXT.md): replace the hardcoded
`static_text: "ESTATE PLANNING"` in `editorial_classic.json` with a
brief-derived value (from `content.tagline`).

**Sections-rendering investigation:** the perception report shows brochures
render with NO sections visible. The root cause may be (a) the JSON kicker
hardcoding (Task 2 fix), (b) the worker passing the wrong dict to the
renderer in `flyer_generator/api/tasks/brochure.py`, or (c) the renderer not
iterating `content.sections` in `flyer_generator/brochure/schema_renderer/renderer.py`.
Task 1 begins with an explicit investigation of (b) and (c) — the JSON-only
fix in Task 2 is conditional on the investigation showing the rendering path
is intact.

Verify with a regression test that the supplied sections actually appear in
the rendered output, and confirm at the WORKER layer that `generate_images: true`
continues to invoke Comfy when `hero_concept` is supplied (patching
`ComfyClient.generate` directly, not `image_gate.generate_template_images` —
the latter is too high in the call stack to detect a worker-side wiring break).

Purpose: A user POSTing `content={..., tagline: "Spring Update", sections: [...]}`
expects the front-cover kicker to read "SPRING UPDATE" (uppercased per the
template), and the three sections to render on inner_left / inner_center /
inner_right. Today the kicker is always "ESTATE PLANNING" because the schema
hardcodes it as a static_text element, AND the perception loop indicates
sections are missing — root cause to be confirmed in Task 1.

Output:
- `flyer_generator/brochure/schemas/editorial_classic.json` updated to remove
  the hardcoded "ESTATE PLANNING" string and instead pull from `content.tagline`
- **Conditional**: `flyer_generator/api/tasks/brochure.py` and/or
  `flyer_generator/brochure/schema_renderer/renderer.py` patched IF the

  patching `ComfyClient.generate` and asserting it is awaited when
  `generate_images: true` is on the request (real bug surface)

DO NOT add a new `kicker` field to BrochureContent. We use `content.tagline`,

empty-text behavior leaves the kicker blank rather than printing the literal
"ESTATE PLANNING" — which is the correct fix.

How the renderer resolves text content (`flyer_generator/brochure/schema_renderer/renderer.py`):
```python
def _resolve_text_content(el: TextElement, content: BrochureContent) -> str:
    if el.role == "static" or el.content_key is None:
        return el.static_text or ""
    raw = content.resolve_key(el.content_key, el.section_index)
    if raw is None:
        return ""
    if isinstance(raw, list):
        return ", ".join(str(item) for item in raw)
    return str(raw)
```

Notice: when `role == "static"`, the renderer returns `static_text` and IGNORES `content_key`. So switching the role from `"static"` to a content-driven role (e.g. `"tagline"`) is required. The role-to-typography map already supports `"tagline"`:
```python
_ROLE_FONT_SIZE = { ..., "tagline": "cover_subtitle_size", ... }
_ROLE_LINE_HEIGHT = { ..., "tagline": 1.2, ... }
```

`content.resolve_key("tagline")` is supported (`flyer_generator/brochure/schema_renderer/content_model.py`):
```python
if key == "tagline":
    return self.tagline
```

`BrochureContent.tagline: str | None = None` (already exists, no model change needed).
