## Summary

Phase 24.1 is a four-bug-fix phase from a perception-loop adversarial run targeting:

- **PLF-01** — postcard worker silently skipped Comfy and emitted a `[ hero ]` placeholder. Fix wires a new `generate_postcard_hero` seam through the canonical `generate_single_image` helper; the postcard renderer now accepts an optional `images={"hero": bytes}` kwarg with full back-compat for legacy positional callers.
- **PLF-02** — brochure `editorial_classic.json` had a hardcoded `static_text: "ESTATE PLANNING"` kicker. Fix swaps it for `content_key: "tagline"` and adds worker-layer tests that patch `ComfyClient.generate` directly (the lowest seam).
- **PLF-03** — `bold_modern.json` headline bbox (h=600 starting at y=120) overlapped the details band (y=1300). Fix caps headline at h=480 and adds a deterministic bbox-collision test plus a sha256 tripwire on the other 5 flyer schemas.
- **PLF-04** — vision evaluation shipped 5400×7200 PNGs to the LLM (≥25 MB base64), causing httpx WriteTimeout. Fix introduces `_downsample_for_vision` (max long edge 1920) applied locally before base64 encoding, preserving `GeneratedBackground.image_bytes` for downstream composer/rasterizer.

**Verifications performed:**

- **Comfy seam (postcard.py):** No leaked secrets — all credentials flow through `Settings`/`SecretStr` via the brochure helper (`generate_single_image`). Worker correctly handles three execution paths: `image_hint=None` (skip), `ctx_http_client` provided (use shared client), and missing client (open an ephemeral 180s `httpx.AsyncClient`). Failure path is best-effort: `except Exception` is logged with `error_type` and falls back to gradient placeholder, exactly matching the `_gen_spot` policy in the brochure pipeline. Local import of `generate_single_image` is appropriately documented and avoids circular imports.
- **Vision downsample (`vision.py`):** `_downsample_for_vision` correctly caps long edge ≤1920px, returns identical bytes on no-op, re-encodes as PNG (preserving the hardcoded `media_type: "image/png"`), and is failure-safe (catches `OSError`/`ValueError` and returns the original bytes). Critically, it operates on a *local copy* in `_evaluate_with_text` — `background.image_bytes` is never mutated. Test `test_evaluate_does_not_mutate_background_image_bytes` verifies both byte-equality and `id()` identity.
- **Mock discipline (worker tests):** Worker tests patch `flyer_generator.stages.comfy_client.ComfyClient.generate` directly via `patch.object(ComfyClient, "generate", new=AsyncMock(...))` — the lowest, real seam. This rules out a regression where a wrapper symbol gets auto-stubbed and the test passes without ever exercising the Comfy call chain. Both positive (`await_count >= 1`) and negative (`await_count == 0` / `assert_not_awaited()`) assertions are present.
- **Schema integrity:** All 5 non-`bold_modern` flyer schemas hash-match the tripwire baseline (verified via `sha256sum`). `bold_modern.json` cover_title bbox is now `[60, 120, 960, 480]` (h=480) with details starting at y=1200 — bottom of title (y=600) clears details by exactly 600px gap, satisfying both the h≤480 cap and the ≥240 gap requirement. `editorial_classic.json` line 33 uses `content_key: "tagline"`; no remaining `"ESTATE PLANNING"` in any brochure schema.

The four fixes are surgically scoped, well-tested, and back-compat-preserving. The findings below are minor maintainability concerns; no Critical issues.

## Warnings

### WR-01: Postcard worker can leak ephemeral httpx client on error

**File:** `flyer_generator/api/tasks/postcard.py:181-199`
**Issue:** When `ctx_http_client is None`, the worker opens an `httpx.AsyncClient` via `async with` *only* for the `generate_postcard_hero` call. That is correct in isolation, but the surrounding `try/except` (line 147 → line 276) means that if `generate_postcard_hero` raises an exception that escapes the helper's own `except Exception` (e.g. `asyncio.CancelledError`, which is intentionally NOT caught by `BLE001` blanket handlers and propagates), the `async with` block correctly unwinds the client. So no leak in normal control flow.

The latent issue is different: `generate_postcard_hero` swallows *every* exception (line 103, `except Exception`), so the worker will *always* succeed past the Comfy step regardless of failure mode. This means a transient `CancelledError` during shutdown cannot propagate, and the postcard will commit a gradient-placeholder render that the user paid for. The brochure spot path has the same policy, so this is intentional and the worker docstring documents it ("postcards must always render"). Document the trade-off explicitly in `image_gate.py:103` so future readers know this swallows `CancelledError` too.

**Fix:** Tighten the `except Exception` to exclude `asyncio.CancelledError`, or add a comment justifying why cancellation is also swallowed:

```python
except asyncio.CancelledError:
    raise  # do NOT swallow shutdown cancellation
except Exception as err:  # noqa: BLE001 — best-effort hero generation
    log.warning("postcard_hero_generate_failed", ...)
    return {}
```

### WR-02: `_resolve_font` swallows `el.role` AttributeError silently when role is None

**File:** `flyer_generator/postcard/schema_renderer/renderer.py:130-154`
**Issue:** `_resolve_font` uses `hasattr(el, "role")` to gate role lookup, but `BulletsElement` has no `role` field at all (only `TextElement` does). On a `BulletsElement`, `hasattr(el, "role")` returns False, so the code falls through to `schema.typography.bullet_size` / `bullet_line_height`. That is the correct branch — but the implicit `hasattr` check is fragile: if a future schema adds a `role` field to `BulletsElement` for any reason (e.g. for a debug field), the resolution order silently changes. Prefer an explicit `isinstance(el, TextElement)` gate.

Also note line 132's role tuple `("cover_title", "section_heading", "cta_heading", "org_name", "quote")` excludes `tagline` — the `editorial_classic.json` cover-front kicker on line 33 is `role: "tagline"` and gets the body family by default (`'Source Serif Pro', Georgia, serif`), which is then overridden by the schema's explicit `font_family: "'Source Serif Pro', Georgia, serif"`. Today this happens to be correct (intentional override), but for any future schema that omits the explicit `font_family` on a tagline element, the heading family will silently NOT be applied. Worth documenting in a comment or adding `tagline` to the heading set if that's the intended default.

**Fix:**

```python
# Replace hasattr checks with explicit type checks
if isinstance(el, TextElement) and el.role in (
    "cover_title", "section_heading", "cta_heading",
    "org_name", "quote", "tagline",  # add tagline if heading family is intended
):
    family = schema.typography.heading_family
```

```python
        prompt_id="fake-prompt-id",

    )
```

### IN-04: Postcard front headline/body bboxes are tightly stacked

**File:** `flyer_generator/postcard/schemas/classic_portrait.json:28-29`
**Issue:** `cover_title` bbox is `[80, 1340, 1040, 220]` (bottom y=1560); `body` bbox is `[80, 1580, 1040, 190]` (top y=1580). Gap = 20px. Given a 96-pt cover_title with a 4-line wrap at line_height ≈ 1.05×96 ≈ 100px, a long headline can render up to ~400px tall, far exceeding the 220px bbox; `fit_to_bbox` will truncate, but if a future schema relaxes the headline cap or the user provides a single-line short headline with a tall ascender, vertical alignment in `valign=top` lands near the top of the bbox (good). However, the same template's `back` panel `body` bbox `[80, 80, 1040, 900]` is much more generous. Consider adding a unit test analogous to PLF-03's `test_no_headline_details_overlap` for postcards to lock the constraint deterministically.
**Fix:** Either widen the gap to ~80px (move body to y=1640) or add a postcard-specific bbox-collision test mirroring `tests/flyer/schema_renderer/test_template_layouts.py::test_no_headline_details_overlap`.

### IN-05: `del template` in `generate_postcard_hero` is a stale-doc smell

**File:** `flyer_generator/postcard/schema_renderer/image_gate.py:74-77`
**Issue:** `del template` followed by a long comment explaining that `template` is unused but kept for forward-compat. The comment is good, but `del` is a slightly unusual idiom — typing `_template = template; del _template` adds nothing. A simpler convention is to rename the kwarg to `_template` (a leading underscore on parameters is a recognised "unused" signal in Python). This avoids the runtime `del`.
**Fix:**

```python
async def generate_postcard_hero(
    template: PostcardTemplateSchema,  # noqa: ARG001 — reserved for future per-template workflow override
    content: PostcardContent,
    ...
```

Or rename to `_template`.
