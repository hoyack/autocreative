# Phase 24.1 — Perception Loop Fixes — Context

This phase fixes 4 cross-phase bugs surfaced by an end-to-end perception loop that exercised every asset-creation API on 2026-04-25 with adversarial Claude vision review. Full report: `/tmp/perception/PERCEPTION-REPORT.md`.

The bugs span Phase 22 (flyer template), Phase 23 (postcard renderer), Phase 23/brochure (brochure renderer), and Phase 24 (poster vision pipeline). They were not caught by existing unit tests because the tests mock either the renderer or the vision provider.

## Bugs in scope

### PLF-01 — Postcard renderer never composes body text or invokes Comfy

**Symptom:** `POST /api/v1/postcards` with `body="Garden Tour..."` and `image_hint="lush spring garden..."` produces a 3-second job that "succeeds" with a render showing only the headline and the literal placeholder string `[ hero ]`. The body text is **completely absent** from the rendered output and no Comfy call is made.

**Evidence:**
- `/tmp/perception/postcard-attempt1-render.png` (classic_portrait): "Save the Date" + `[ hero ]` placeholder, no body
- `/tmp/perception/postcard-attempt2-render.png` (modern_landscape): same shape, different layout
- Worker log shows no `comfy_submitted` line for postcard tasks
- Switching templates does not help — bug is in the renderer

**Root-cause hypothesis:** The postcard task `flyer_generator/api/tasks/postcard.py` calls `render_postcard(template, content)` directly without first generating a Comfy background. The schema renderer `flyer_generator/postcard/schema_renderer/renderer.py` likely has a `[ hero ]` placeholder fallback for when no image is supplied, and the body-text render path may be unwired.

**Acceptance:**
- Postcard task generates a Comfy background from `image_hint` (mirroring the brochure pipeline pattern when `generate_images: true`) before calling `render_postcard`
- `body` text renders visibly on the front PNG for both `classic_portrait` and `modern_landscape`
- `[ hero ]` placeholder no longer appears in any rendered postcard front
- New pytest `tests/postcard/test_render_smoke.py::test_postcard_renders_body_and_hero_image` asserts both via Pillow inspection (or a layout-coordinate check)
- 9-permutation HTTP test still passes

### PLF-02 — Brochure renderer drops content.sections and ships hardcoded "ESTATE PLANNING" kicker

**Symptom:** `POST /api/v1/brochures` with `content.sections=[Our Vision, By the Numbers, Looking Forward]` and `generate_images: true` produces a render where:

1. The hero panel shows literal `[ hero ]` placeholder (no Comfy image despite the flag)
2. The kicker reads "ESTATE PLANNING" — that string was not in the brief; it appears to be a default/template-baked value
3. None of the 3 supplied content sections appear anywhere in the rendered output
4. Title + subtitle + tagline DO render correctly (so partial paths work)

**Evidence:**
- `/tmp/perception/brochure-attempt1-render.png` (editorial_classic): all 3 issues visible

**Root-cause hypothesis:** Either the brochure task is passing an empty/wrong dict to `render_brochure`, OR the brochure renderer (`flyer_generator/brochure/schema_renderer/renderer.py`) is not reading `content.sections` and is using a hardcoded fallback. The "ESTATE PLANNING" string is likely a template default that fires when the relevant content field is empty / unmapped. The `generate_images` flag is not triggering Comfy.

**Acceptance:**
- Each supplied `ContentSection.heading` appears as a visible heading in the rendered front PNG
- Each `ContentSection.body_paragraphs` text appears as visible body copy under its heading
- Kicker text is derived from the brief (e.g., from `content.tagline` or a new explicit field) — not hardcoded "ESTATE PLANNING"
- `generate_images: true` produces a Comfy-generated image in the hero panel; `generate_images: false` falls back to a clean placeholder (NOT the literal text `[ hero ]`)
- New pytest in `tests/brochure/` asserts all 3 conditions

### PLF-03 — Flyer `bold_modern` template has a headline-vs-detail-text z-order collision

**Symptom:** `POST /api/v1/flyers` with `template="bold_modern"` produces a render where the headline ("SPRING TECH MEETUP") and the date/time/venue text ("2026-05-20", "6:30pm", "The Fillmore", "1805 Geary Blvd...") visibly **overlap** in the same vertical band. Letters intersect, producing a visual mess that the adversarial vision flagged as "garbled / corrupted text rendering."

**Evidence:**
- `/tmp/perception/flyer-attempt2-render.png` (bold_modern): "MEETUP" letters punch through "The Fillmore" line; "2026-05-20" sits inside "T E C H"

**Root-cause hypothesis:** The `bold_modern` flyer template (`flyer_generator/flyer/schemas/bold_modern.json`) has overlapping y-coordinate bands for the headline element and the details block. Other templates (`editorial_classic`) are spaced correctly. Likely a template-data fix, not a renderer-code fix.

**Acceptance:**
- Re-render with the same input on `bold_modern` produces no visual overlap between headline and details
- A new pytest synthesizes the same flyer at the worker layer (mocking Comfy) and asserts that the headline bounding box and the details bounding box do not intersect (purely deterministic — no AI in the test path)
- The other 5 flyer templates remain unchanged

### PLF-04 — Poster vision pipeline times out because vision input is the full 5400×7200 PNG

**Symptom:** `POST /api/v1/posters` consistently fails with `LLMAPIError: WriteTimeout` (against either Ollama Cloud or Anthropic). The job runs Comfy successfully, then dies during the vision-evaluation upload because the upscaled background is 5400×7200 (or larger for 24×36 / 27×40), encoding to ~25–50MB of base64 in the request body. Even a 180-second timeout × 3 retries (~540s) is not enough on a typical residential WSL2 connection.

**Evidence:**
- `/tmp/arq-24.log` traces show `httpx.WriteTimeout` inside `_call_anthropic` → `messages.create`
- Failed jobs: `01KQ28D29MH9E2NDDT3G5X7VHV` (ollama, 7min), `01KQ29WJMBV957S183DYEA7EB4` (anthropic, 213s), `01KQ2AM9EE6R3REJ25T67B26GN` (anthropic-180s, 571s)
- `flyer_generator/pipeline.py:138-141`: `background = self._preprocessor.upscale(raw_bytes, comfy_job)` then `verdict = await self._vision.evaluate(background, event)` — vision sees the upscaled, full-canvas image
- This bug was not caught by tests because `tests/poster/test_render_smoke.py` mocks the vision call

**Root-cause hypothesis:** `Vision.evaluate` doesn't need full canvas resolution — it picks zone labels, not pixel-precise coordinates. Sending a downsampled thumbnail (e.g., long-edge ≤1920px) is functionally equivalent and ~30× smaller.

**Acceptance:**
- A new helper (in `vision.py` or a small util) downsamples the image to a max long-edge of 1920px before base64 encoding for the LLM call (preserve original `image_bytes` for the rest of the pipeline)
- `POST /api/v1/posters` with size=18×24 completes end-to-end against real Anthropic vision in <120 seconds on a typical residential connection
- All 3 poster sizes (18×24, 24×36, 27×40) succeed with `FLYER_VISION_TIMEOUT_SECONDS=60`
- Existing 9 render-smoke pytests still pass (they mock vision so they won't notice the change, but ensure no regressions)
- New pytest verifies the downsampling is applied (e.g., spy on the bytes passed to `_call_anthropic` and assert size after decode is ≤1920px on long edge)
