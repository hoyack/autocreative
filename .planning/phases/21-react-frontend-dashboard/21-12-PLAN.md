<objective>
Close two review warnings in the brochure subsystem:

**WR-01**: `flyer_generator/api/tasks/brochure.py:78` reads `payload["workflow_name"]`, but `BrochureCreateRequest` (schemas/brochures.py:24) defines the field as `workflow`. The POST route serializes via `body.model_dump(mode="json")` which produces the key `"workflow"`, so any user-supplied override is silently dropped and the hard-coded `"turbo_landscape"` is always used.

**WR-03 (brochures half)**: `flyer_generator/api/routes/brochures.py:32-47` commits the JobRecord before calling `arq_pool.enqueue_job`. If enqueue raises (Redis down, arq pool misconfigured, etc.), the JobRecord stays `QUEUED` forever with no worker to advance it. The dashboard's `/jobs` list shows a ghost row and status polling spins forever.

Purpose: Make user-supplied brochure workflow overrides actually take effect, and make failed enqueues surface as a visible FAILED JobRecord rather than a stale QUEUED ghost.

Output: One-line key fix in the worker + try/except wrap in the route + two regression tests (one per warning).
</objective>

    ## Step 2 (GREEN) — wrap enqueue in try/except compensating transition.

    Edit `flyer_generator/api/routes/brochures.py`. Replace the existing `create_brochure` body (lines 24-47) — specifically the block that does `await session.commit()` then `await request.app.state.arq_pool.enqueue_job(...)` — with the compensating-transition pattern:

    ```python
    from flyer_generator.api.models import JobStatus  # ADD to imports if not present

    @router.post(
        "/brochures",
        status_code=status.HTTP_202_ACCEPTED,
        response_model=JobCreated,
        summary="Enqueue a brochure generation (3 artifacts: front PNG + back PNG + PDF)",
    )
    async def create_brochure(
        body: BrochureCreateRequest,
        request: Request,
        session: AsyncSession = Depends(get_session),
    ) -> JobCreated:
        job_id = str(ulid.ULID())
        payload = body.model_dump(mode="json")

        session.add(
            JobRecord(
                id=job_id,
                kind=JobKind.BROCHURE,
                status=JobStatus.QUEUED,
                input_payload=payload,
            )
        )
        await session.commit()

        # WR-03: compensate on enqueue failure. If arq can't accept the job
        # (Redis down, pool misconfigured, etc.), the QUEUED row above has no
        # worker to advance it — flip it to FAILED before the 5xx propagates
        # so the dashboard reflects reality instead of showing a ghost.
        try:
            await request.app.state.arq_pool.enqueue_job(
                "task_generate_brochure",
                job_id=job_id,
                payload=payload,
            )
        except Exception as exc:
            # Use a fresh session: the request-scoped `session` may be in a
            # dirty state depending on the failure path. The error_detail is
            # deliberately minimal ({"reason": "enqueue_failed", "type": ...})
            # to avoid leaking Redis connection strings or stack frames into
            # the DB column (T-21-12-03).
            async with request.app.state.sessionmaker() as s2:
                row = await s2.get(JobRecord, job_id)
                if row is not None:
                    row.status = JobStatus.FAILED
                    row.error_detail = {
                        "reason": "enqueue_failed",
                        "type": type(exc).__name__,
                    }
                    await s2.commit()
            raise

        return JobCreated(job_id=job_id)
    ```

    Notes for the executor:
    - `request.app.state.sessionmaker` is bound in `app` lifespan AND in tests/conftest.py (line 77: `app.state.sessionmaker = sessionmaker_fx`). Do NOT re-use the dependency-injected `session` — the request transaction may already be committed/closed.
    - Do NOT write `str(exc)` into `error_detail` — the reviewer explicitly called out that error_detail must NOT leak connection strings. Use `type(exc).__name__` only.
    - The re-raise lets FastAPI's exception handler surface a 500 with the generic `{detail, error_type, trace_id}` body.
