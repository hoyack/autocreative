# Phase 24.2 Plan 02: RM-02 Render Delete Capability Summary

DELETE /api/v1/renders/{render_id} ships end-to-end: backend route nulls FK columns on five parent tables (postcards, brochures, posters, flyers, posts) via explicit UPDATE statements, deletes the RenderRecord row, best-effort-unlinks the on-disk artifact with the same containment guard as get_render_image, and returns 204; the renders gallery now sports a Trash2 button on every card that opens an AlertDialog and fires an optimistic-update DELETE mutation.

## What Shipped

### Backend

- **DELETE /api/v1/renders/{render_id}** in `flyer_generator/api/routes/renders.py`:
  - Returns **204 No Content** on success.
  - Returns **404 Not Found** when the row is unknown OR was already deleted (re-delete is NOT silent — it surfaces as 404 per the CONTEXT.md decision).
  - Iterates a tuple of `(model, FK-cols)` pairs and issues `UPDATE {model} SET {col}=NULL WHERE {col}={render_id}` for every render-pointing FK column on every parent table:
    - `PostcardRecord.{render_front_id, render_back_id, render_pdf_id}`
    - `BrochureRecord.{render_front_id, render_back_id, render_pdf_id}`
    - `PosterRecord.render_id`
    - `FlyerRecord.render_id`
    - `PostRecord.render_id`
  - Captures `file_path` BEFORE `session.delete(record)` (avoiding stale-attribute UB on the detached instance).
  - Calls `os.unlink(file_path)` only when the resolved path lives inside one of the four configured artifact roots (`artifact_root_flyer`, `artifact_root_brochure`, `brand_kits_dir`, `social_campaigns_dir`) — same `_is_within` guard used by `get_render_image`.
  - Catches `FileNotFoundError` and `OSError` from unlink → logs a structured warning (`render_delete_file_missing` / `render_delete_unlink_failed`), still returns 204.
  - Path-outside-roots → logs `render_delete_path_outside_roots`, skips unlink, still returns 204 (row deleted; file is not ours to touch).

### Frontend

- **`frontend/src/components/ui/alert-dialog.tsx`** — new shadcn-style wrapper around radix-ui's `AlertDialog` primitive. Mirrors `dialog.tsx` visually but enforces explicit Action/Cancel choice (no Close X), and `AlertDialogCancel` is the canonical default-focus per Radix accessibility convention.
- **`frontend/src/pages/renders/gallery.tsx`**:
  - Per-card Trash2 ghost-button at `top-2 right-2` with `bg-background/70` so it remains visible against the artwork.
  - `<Button onClick={(e) => e.stopPropagation()}>` keeps clicks from bubbling into the PDF download anchor.
  - AlertDialog with `Title="Delete render?"`, destructive-variant Action button, and a Cancel button.
  - TanStack `useMutation`:
    - `mutationFn` calls `client.DELETE("/api/v1/renders/{render_id}", { params: { path: { render_id } } })` (auto-derived from regenerated `schema.gen.ts`).
    - `onMutate` cancels in-flight queries, snapshots the page, filters the deleted card out of the cache, returns the snapshot for rollback.
    - `onError` restores from snapshot and surfaces `toast.error(err.message)`.
    - `onSuccess` calls `toast.success("Render deleted")` and `queryClient.invalidateQueries({ queryKey: queryKeys.renders() })` (bare key form for prefix-match invalidation across every paginated/filtered variant).
- **OpenAPI snapshot** (`frontend/src/api/openapi.snapshot.json`) regenerated in-process via `build_app().openapi()` (Phase 24-05 pattern); `frontend/src/api/schema.gen.ts` regenerated via `pnpm gen:api:snapshot` so the DELETE operation gets a typed signature (`delete_render_api_v1_renders__render_id__delete`).
- **`frontend/src/api/client.ts`** — comment block notes the DELETE availability; no hand-written typed alias needed (openapi-fetch derives it from the paths type).

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Re-delete returns 404, not silent 204 | Explicit not-found beats hidden idempotency; surfaces UI/test bugs that would otherwise stay invisible |
| Explicit UPDATE-then-DELETE for FK nulling | SQLite has `PRAGMA foreign_keys=OFF` in `flyer_generator/api/db.py` so schema-level `ondelete="SET NULL"` does not fire; explicit UPDATE makes the route DB-engine-agnostic |
| AlertDialog over Dialog | Destructive actions need explicit Action/Cancel choice — no dismissable Close X; default focus on Cancel prevents accidental Enter-press deletion |
| Path-outside-roots logged but row still deleted | User intent is "purge from gallery"; refusing to unlink an unknown-path file is correct (containment) but row deletion is what the user asked for |
| No new npm dependency | radix-ui umbrella package re-exports `react-alert-dialog` as `AlertDialog` (verified in `node_modules/radix-ui/src/index.ts`) — `lucide-react` and `sonner` were already in the lockfile |
| Stateful msw GET in optimistic-remove test | The post-success `invalidateQueries` triggers a refetch; if the GET handler keeps serving the original list, the optimistically-removed card reappears. Tracking `currentItems` in the handler reflects reality |

## Lessons

- **SQLite tests need explicit UPDATE for FK nulling.** The schema-level `ondelete="SET NULL"` directives on every parent ForeignKey only fire when `PRAGMA foreign_keys=ON` — which this codebase does NOT enable (no `listens_for("connect")` toggle in `db.py`). Without an explicit UPDATE, SQLite would happily leave dangling render_*_id pointers after the row is gone, corrupting the detail-route fuse. Postgres production would fire the cascade automatically; writing the UPDATE explicitly removes the DB-engine surprise.
- **radix-ui umbrella package is the integration story.** `node_modules/radix-ui/src/index.ts` re-exports every `@radix-ui/react-*` primitive as a named export (`AlertDialog`, `Dialog`, `DropdownMenu`, ...). Existing components import `Dialog as DialogPrimitive` from this umbrella; mirroring the pattern for `AlertDialog as AlertDialogPrimitive` works without touching `package.json`.
- **TanStack optimistic update + invalidate needs realistic msw state.** If the GET handler keeps serving the original list, the post-`invalidateQueries` refetch resurrects the optimistically-removed row. Tracking the deletion in the msw handler closure mirrors what the real backend does after a successful 204.
- **Phase 24-05's in-process OpenAPI regen pattern is now the project default.** `from flyer_generator.api import build_app; spec = build_app().openapi()` writes the snapshot without needing a running uvicorn — critical for parallel-worktree agents that can't safely contend with the host's dev server.

## Deviations from Plan

None of significance. Three minor adjustments worth noting:

1. **Test #3 used a stateful msw list, not a stub-on-DELETE.** The plan suggested asserting the request fired and the card disappeared post-mutation. In practice the `onSuccess` invalidate-and-refetch race meant a static GET handler resurrected the optimistically-removed row. Tracking `currentItems` in the handler closure (mirrors real backend state) made the test reflect the actual user experience.
2. **Trash button uses `size="icon-sm"` (32px), not the plan's `size="icon"` (40px).** Visual fit on a 200px-wide thumbnail card needed the smaller variant; the existing `dialog.tsx` Close button already uses `icon-sm` for the same reason.
3. **The plan suggested `frontend/src/api/client.ts` get a comment "near the existing aliases".** Placed it as a "Mutation availability notes" block before the Response aliases section; same intent, slightly different position.
