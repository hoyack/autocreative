## RM-01 — Print PDF page-size bug (postcard + brochure)

### Symptom

Live evidence:
- Brochure PDF `01KQ2Y16WB3DEMDC5QHFZSM9R6`: `pypdf` reports mediabox `3376 × 2626 pt` → **46.89 × 36.47 in** (should be 11.25 × 8.75 in)
- Postcard schemas at 300dpi (1200 × 1800 px portrait, 1800 × 1200 px landscape) — PDF would report 16.67 × 25 in or 25 × 16.67 in (should be 4 × 6 in or 6 × 4 in)

### Root cause

Both PDF assemblers pass 300dpi pixel dimensions directly to reportlab's `Canvas(buf, pagesize=(w, h))`. ReportLab interprets `pagesize` in **PostScript points** (1 pt = 1/72 in). At 300 dpi the conversion factor is 72/300 = 0.24 — pixel values are 4.17× too large.

Files:
- `flyer_generator/brochure/stages/pdf.py:81` — `Canvas(buf, pagesize=(BLEED_CANVAS_WIDTH, BLEED_CANVAS_HEIGHT))` where the constants are 3376 × 2626 px (computed in `flyer_generator/brochure/stages/layout.py` from `LETTER_LANDSCAPE_INCHES * 300`)
- `flyer_generator/postcard/stages/pdf.py:88` — `Canvas(buf, pagesize=(page_width, page_height))` where `page_width`/`page_height` come from `PostcardTemplateSchema.canvas` (1200 × 1800 px etc.)
- `flyer_generator/brochure/stages/pdf.py::_draw_crop_marks` — uses `BLEED_PX` (38px) and `_CROP_LEN = 36` (pixel-ish) — must scale together

The postcard pdf.py docstring even explicitly notes "passing pixel values here gives us a 1:1 pixel-to-point page so the caller does not need DPI math" — that is the bug, codified as a "feature."

### Fix

Two places need a 72/300 conversion. Cleanest pattern: `Canvas.scale(PT_PER_PX, PT_PER_PX)` after construction, with `pagesize` in points; subsequent `drawImage` and crop-mark coordinates remain in pixel units (no other code changes). Postcard helper takes pixel dims today — keep that interface but convert internally.

```python
PT_PER_PX = 72.0 / 300.0  # 0.24

page_w_pt = page_width_px * PT_PER_PX
page_h_pt = page_height_px * PT_PER_PX
canvas = Canvas(buf, pagesize=(page_w_pt, page_h_pt))
canvas.scale(PT_PER_PX, PT_PER_PX)  # user-space stays in 300dpi pixels
canvas.drawImage(..., 0, 0, width=page_width_px, height=page_height_px, ...)
# crop marks at pixel coordinates work unchanged
```

## RM-02 — Render delete capability

### Symptom

User has many renders accumulating in the gallery; no UI affordance to delete experimental ones. Only purge mechanism today is direct DB + filesystem manipulation.

### Backend

- `DELETE /api/v1/renders/{render_id}` route — new
- Returns `204 No Content` on success
- Returns `404 Not Found` if render does not exist (or already deleted — idempotent path)
- Deletes:
  1. The on-disk file (the path stored in `RenderRecord.artifact_path` under `<artifact_root>/.../<render_id>.{png,pdf}`)
  2. The `RenderRecord` row
- Cascade behavior: a `RenderRecord` may be referenced by `PostcardRecord.render_{front,back,pdf}_id`, `BrochureRecord.render_{front,back,pdf}_id`, or `JobRecord.result_ref` (string). Decision: set the FK columns to `NULL` (the parent record stays, the missing render is reflected in the detail route returning `null` for that URL). Don't hard-block on FK; let users delete experimental renders even if a parent record references them.
- Path-traversal hardening: `render_id` already validated by the existing 26-char ULID PathParam guard
- File I/O failure handling: if DB row deletes but disk file is missing or unwriteable, log a warning and still return 204 — the user's intent (purge from gallery) is satisfied
- New alembic migration is NOT required (no schema change — just nulling existing FK columns at runtime)

### Frontend

- `frontend/src/pages/renders/gallery.tsx`:
  - Trash icon button (lucide-react `Trash2`) on each render card, top-right corner with subtle styling so it doesn't dominate
  - Click opens shadcn `AlertDialog` with title "Delete render?", description "This will permanently delete the file. Parent jobs/brochures/postcards keep their record but the artifact will be gone. This cannot be undone."
  - Confirm button is destructive variant (red)
  - On confirm: call `client.deleteRender({ params: { path: { render_id } } })` mutation, on success invalidate the renders list query (TanStack Query `queryClient.invalidateQueries({ queryKey: ['renders'] })`), on error show toast
  - Optimistic UI: immediately remove the card from the gallery view; if the API errors, restore + toast
- `frontend/src/api/openapi.snapshot.json` regenerated to include the DELETE route
- `frontend/src/api/schema.gen.ts` regenerated
- `frontend/src/api/client.ts` — typed alias `DeleteRenderParams` and `client.deleteRender(...)` if not auto-generated
- vitest covers: dialog opens on trash click, confirm triggers mutation, optimistic removal works, error path restores card

### Acceptance (RM-02)

## Decisions locked

- 300dpi → 72dpi conversion factor lives as a module-level constant `PT_PER_PX = 72.0 / 300.0` in each pdf.py (not extracted to a shared module — the conversion is local and simple)
- Render delete sets FK columns to NULL rather than hard-blocking — favors user agency over referential strictness (no audit-log requirement today)
- AlertDialog confirmation, not a one-click trash — accidental deletion costs the user a regenerate run
- No soft-delete / trash bin (out of scope; would add a `deleted_at` column + a separate route to undelete — not justified for v1.1)
- Frontend uses TanStack Query's optimistic update + rollback-on-error pattern (matches existing codebase patterns)
